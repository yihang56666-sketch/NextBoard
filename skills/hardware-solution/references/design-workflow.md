# Hardware Solution Design Workflow

## 1. 需求冻结

先把需求整理成工程约束表。缺失项用“待确认”，不要默认补齐。

- 场景：安装位置、用户交互、工作时长、维护方式。
- 环境：温度、湿度、振动、ESD、EMI、户外/工业/车载/医疗等边界。
- 供电：电池、USB、适配器、PoE、车电、宽压输入、隔离需求。
- 功能：采集、控制、显示、音频、图像、联网、存储、安全。
- 量产：目标成本、年用量、认证、测试夹具、可维修性。

## 2. 架构候选

至少给出两个合理方案，除非用户明确指定平台。

- 低成本 MCU 方案：适合控制、采集、低功耗、简单 UI。
- 高性能 MCU/MPU 方案：适合图形、音频、协议栈、边缘计算。
- 模组化方案：适合快速上市、无线认证压力大、研发资源有限。
- 分立方案：适合成本敏感、量产规模大、需要深度定制。

每个候选方案都要比较：BOM 成本、开发周期、供应风险、功耗、PCB 难度、软件复杂度、认证风险、可扩展性。

## 3. 系统分解

按功能域拆硬件，不要按器件堆列表。

- 主控与存储：MCU/SoC、Flash、RAM、启动方式、调试接口。
- 电源树：输入保护、充电/电源路径、DCDC/LDO、负载开关、采样。
- 通信：USB、UART、CAN、RS485、Ethernet、Wi-Fi、BLE、LTE、LoRa。
- 传感与执行：模拟前端、ADC/DAC、隔离、保护、校准。
- 人机交互：按键、LED、蜂鸣器、屏幕、触摸、音频。
- 安全与可靠性：看门狗、掉电保护、ESD/浪涌、热设计。
- 生产测试：测试点、烧录口、边界扫描、校准夹具、序列号写入。

## 4. 关键选型

对每个关键器件给出“选择理由 + 替代方案 + 风险”。

- MCU/SoC：资源余量至少覆盖当前需求，并说明 GPIO、RAM、Flash、外设、封装、功耗、生态。
- 电源芯片：输入范围、效率、热耗散、纹波、EMI、保护特性、封装可焊性。
- 无线器件：频段、天线、认证、射频布局、吞吐、功耗、供应区域。
- 连接器：插拔寿命、防呆、锁扣、线束成本、现场维护。
- 存储：擦写寿命、掉电一致性、文件系统或 wear leveling 需求。

## 5. 约束输出

方案必须产出能指导原理图和 PCB 的约束。

- 接口矩阵：信号名、方向、电平、电流、速率、保护、连接器针脚。
- 电源树：输入、保护、电源轨、电流预算、上电时序、关断策略。
- PCB 约束：层数建议、关键走线、阻抗、回流路径、模拟/数字/射频分区。
- 机械约束：板框、安装孔、连接器方向、散热、天线净空。
- DFM/DFT：测试点、烧录方式、工装定位、生产校准项。

## 6. 验证计划

每个风险都要有验证动作。

- EVT：核心功能、电源稳定性、接口连通、基本环境测试。
- DVT：边界条件、EMC 预扫、温升、ESD、长稳、异常掉电。
- PVT：量产测试覆盖率、工装节拍、不良定位、替代料验证。

## 7. 决策门

在输出方案末尾明确三类事项。

- 已确定：可以进入原理图设计的内容。
- 待确认：需要用户、结构、采购、认证或软件团队补充的信息。
- 高风险：若不验证会影响成本、周期、可靠性或量产的内容。
