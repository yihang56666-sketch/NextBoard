# Hardware Solution Design Workflow

> **文件输出规则：** 有表格产出的阶段必须输出为 markdown 文件至 `docs/hardware/` 目录。详见 SKILL.md「文件输出规则」。

## 0. 模式选择

进入设计流程前，先询问用户选择工作模式：

- **快速模式**：对未提供的约束自动按合理假设推进，一次性输出需求表和假设清单供用户审阅，用户可批量确认或修改。跳过逐项问答，快速进入架构和选型。适合原型验证、技术预研、个人项目。
- **专业模式**：严格执行需求冻结，所有"待确认"项逐一确认（每次只问 1 个问题），用户逐项回复后才能推进。适合量产项目、团队协作、正式交付。

模式选择后记录到 `01-requirements.md` 顶部。后续阶段根据模式调整交互方式。

## 1. 需求冻结

先把需求整理成工程约束表。缺失项用“待确认”，不要默认补齐。

- 场景：安装位置、用户交互、工作时长、维护方式。
- 环境：温度、湿度、振动、ESD、EMI、户外/工业/车载/医疗等边界。
- 供电：电池、USB、适配器、PoE、车电、宽压输入、隔离需求。
- 功能：采集、控制、显示、音频、图像、联网、存储、安全。
- 量产：目标成本、年用量、认证、测试夹具、可维修性。
- 供应链偏好：国产优先、海外主流优先、混合方案、无偏好。

需求表之后必须给用户一个结构化选择题，而不是直接深入方案。例如：

- 优先低功耗：牺牲部分成本和开发便利，适合电池长续航。
- 优先低成本：选择国产或高性价比器件，适合明确量产规模。
- 优先快速落地：选择生态成熟器件和参考设计，适合原型或交付周期紧。
- 优先国产供应链：覆盖国产 MCU、电源、接口芯片和传感器替代，适合国产化要求。

**逐项确认规则（专业模式）：** 所有"待确认"项必须逐一向用户确认，用户可选择 (a) 提供确认值 (b) 授权基于假设推进 (c) 标注阻塞。不能默认跳过或自行假设。**每次只问 1 个问题，等用户回复后再问下一个，严禁一次性列出多个问题。每个问题的选项格式：**

```
问题：[具体问题]
A. [选项1]
B. [选项2]
C. [选项3]
D. 自定义输入：___
```

**快速确认规则（快速模式）：** 一次性列出完整需求表，未提供的项填入合理假设并标注"[假设]"。用户可批量确认、逐条修改或回复"按假设推进"。无需逐项问答。

**输出文件：** `docs/hardware/01-requirements.md`

## 2. 架构候选

至少给出三个合理方案，除非用户明确指定平台或明确排除某类供应链。

- 国产芯片/国产供应链优先方案：覆盖国产 MCU、电源、RS485/CAN/接口、存储、保护器件；适合国产化、成本敏感或供应链自主要求。
- 海外主流生态优先方案：覆盖 STM32、TI、NXP、Microchip、ADI、MaxLinear 等成熟生态；适合开发周期紧、参考设计丰富、可靠性优先。
- 混合折中方案：主控、电源、通信、传感器按风险拆分国产和海外器件；适合成本、开发效率、供应安全同时要平衡。
- 模组化方案：适合快速上市、无线认证压力大、研发资源有限。

每个候选方案都要比较：BOM 成本、开发周期、供应风险、功耗、PCB 难度、软件复杂度、认证风险、可扩展性。

不要把"国产/海外"写成口号。每个候选方案必须至少列出主控、电源、通信接口、传感器/模拟前端的候选器件系列或选择范围，并标注需要联网查证的数据。

**输出文件：** `docs/hardware/02-architecture.md`

## 3. 系统分解

按功能域拆硬件，不要按器件堆列表。

- 主控与存储：MCU/SoC、Flash、RAM、启动方式、调试接口。
- 电源树：输入保护、充电/电源路径、DCDC/LDO、负载开关、采样。
- 通信：USB、UART、CAN、RS485、Ethernet、Wi-Fi、BLE、LTE、LoRa。
- 传感与执行：模拟前端、ADC/DAC、隔离、保护、校准。
- 人机交互：按键、LED、蜂鸣器、屏幕、触摸、音频。
- 安全与可靠性：看门狗、掉电保护、ESD/浪涌、热设计。
- 生产测试：测试点、烧录口、边界扫描、校准夹具、序列号写入。

## 4. 关键选型

对每个关键器件给出“选择理由 + 替代方案 + 风险”。

- MCU/SoC：资源余量至少覆盖当前需求，并说明 GPIO、RAM、Flash、外设、封装、功耗、生态。
- 电源芯片：输入范围、效率、热耗散、纹波、EMI、保护特性、封装可焊性。
- 无线器件：频段、天线、认证、射频布局、吞吐、功耗、供应区域。
- 连接器：插拔寿命、防呆、锁扣、线束成本、现场维护。
- 存储：擦写寿命、掉电一致性、文件系统或 wear leveling 需求。

选型阶段必须显式比较国内外候选项：

- 国产候选：说明国产替代价值、生态成熟度、资料可得性、代理/分销渠道、长期供货风险。
- 海外候选：说明生态优势、参考设计、认证资料、价格和交期风险。
- 混合候选：说明哪些模块必须稳妥、哪些模块可以国产化，以及替代验证成本。

每个推荐的关键 IC 和功能模块器件附带采购参考链接，优先级：国内电商平台（淘宝/立创商城/嘉立创） > 授权分销商 > 原厂官网。以方便个人购买为主。同时提供 datasheet 下载链接（优先 AllDatasheet > 立创商城 > 半导小芯 > 原厂官网）和封装文件下载链接（优先立创 EDA 封装库/华秋 DFM 封装库 > 嘉立创封装库 > SnapEDA/Ultra Librarian > 原厂）。无法提供时标注"需联网查证"。

器件分层：
- 关键 IC（需提供链接和资料）：MCU、PMIC、收发器、传感器、ADC/DAC 等核心芯片
- 功能模块器件（需提供链接和资料）：晶振、连接器、ESD/TVS 保护、存储芯片、负载开关、LDO 等
- 被动元件（不需要提供）：电阻、电容、电感等通用规格件，EDA 库中通常已有

**输出文件：** `docs/hardware/03-components.md`

## 4.5 资料下载

Gate 3 通过后，先询问用户使用的 EDA 工具（KiCad / Altium Designer / 立创 EDA / 其他），然后批量下载已选的关键 IC 和功能模块器件的 datasheet 和对应格式的封装文件。被动元件不下载（EDA 库中通常已有）。

**下载经验记录：** 下载前先读取 [download-sources.md](download-sources.md)，优先使用已验证的源。下载成功后将器件型号、源站点、URL、日期追加到该文件。下载失败也记录到"已知失败源"表，避免后续重复尝试同一个失败源。

**Datasheet 下载策略（严格按顺序尝试，禁止跳过前序源）：**

1. AllDatasheet（alldatasheet.com）— 直链 PDF，无反爬，优先使用
2. 立创商城（lcsc.com）器件详情页的 datasheet 链接
3. 半导小芯（semiee.com）— 国内 datasheet 聚合站
4. 原厂官网 — 仅在前三个源都无法获取时才尝试

**反爬/下载失败处理规则：**

- 如果某个源返回 HTML 页面而非 PDF，立即跳过该源，尝试下一个
- 如果某个源需要登录、验证码或动态 JS 渲染，立即跳过，不要重试
- ST、TI、NXP、Mouser、DigiKey 等原厂/分销商站点大概率有反爬限制，应排在最后尝试
- 所有源都失败时，在 `03-components.md` 中标注"需手动下载"并附上最可能成功的页面 URL（优先 AllDatasheet 搜索链接）
- 禁止保存非 PDF 文件（HTML、重定向页面）到 datasheets 目录

**封装文件下载策略（严格按顺序）：**

1. 立创 EDA 封装库 / 华秋 DFM 封装库 — 国内平台，直链下载
2. 嘉立创封装库
3. SnapEDA / Ultra Librarian — 可能需要注册，优先使用免登录直链
4. 原厂 — 最后尝试

封装下载失败时同样标注"需手动下载"并附链接。

- Datasheet（PDF）保存至 `docs/hardware/datasheets/`，文件名格式：`器件型号.pdf`。
- 封装文件保存至 `docs/hardware/footprints/`，格式由用户指定的 EDA 决定：
  - KiCad → `.kicad_mod`
  - Altium Designer → `.PcbLib`
  - 立创 EDA → `.elib`
  - 其他 → 按用户指定格式或通用格式（`.bxl`）
- 下载优先级同选型阶段链接优先级。
- 下载失败的记录原因，并在 `03-components.md` 中标注手动下载链接。
- 下载完成后输出清单，列出每个器件的文件路径和状态（已下载/失败/需手动）。

**输出目录：** `docs/hardware/datasheets/` 和 `docs/hardware/footprints/`

## 5. 约束输出

方案必须产出能指导原理图和 PCB 的约束。

- 接口矩阵：信号名、方向、电平、电流、速率、保护、连接器针脚。
- 电源树：输入、保护、电源轨、电流预算、上电时序、关断策略。
- PCB 约束：层数建议、关键走线、阻抗、回流路径、模拟/数字/射频分区。
- 机械约束：板框、安装孔、连接器方向、散热、天线净空。
- DFM/DFT：测试点、烧录方式、工装定位、生产校准项。

**输出文件：** `docs/hardware/04-constraints.md`

## 6. 验证计划

每个风险都要有验证动作。

- EVT：核心功能、电源稳定性、接口连通、基本环境测试。
- DVT：边界条件、EMC 预扫、温升、ESD、长稳、异常掉电。
- PVT：量产测试覆盖率、工装节拍、不良定位、替代料验证。

**输出文件：** `docs/hardware/05-validation.md`

## 7. 决策门

在输出方案末尾明确三类事项。

- 已确定：可以进入原理图设计的内容。
- 待确认：需要用户、结构、采购、认证或软件团队补充的信息。
- 高风险：若不验证会影响成本、周期、可靠性或量产的内容。

**输出文件：** `docs/hardware/06-decisions.md`

## 8. 模块原理图输出

评审通过后询问用户是否需要输出模块原理图。用户确认需要后，提供展示方式选项：

- 结构化连接表：网络名、源端器件.引脚、目标端器件.引脚、参数/备注，精确无歧义，适合直接交给工程师。无需额外工具。
- ASCII 框图：用文本符号表达电路拓扑和电源路径，标注器件型号与关键参数值，直观易读，无需工具。
- D2 图表（.d2）：用 D2 语法描述模块层级和连接关系，支持嵌套和连线标注，渲染后可视化效果好。需安装 d2 CLI 渲染为 SVG/PNG。
- KiCad 原理图文件（.kicad_sch）：**需要 KiCad MCP Server 支持。** 通过 MCP 工具调用 KiCad API 完成器件放置、连线和标注，生成可直接用 KiCad 打开的原理图文件。
- 立创 EDA JSON：**需要 JLCEDA MCP Server 支持。** 通过 MCP 工具调用立创 EDA API 生成可直接导入的原理图数据，适合国内用户和立创 EDA 工作流。

**MCP Server 依赖说明：**

KiCad 和立创 EDA 格式依赖对应的 MCP Server 在当前环境中可用。LLM 不直接生成原始 .kicad_sch 或 JSON 文件（格式复杂、坐标精度要求高、符号库依赖强），而是通过 MCP 工具调用 EDA 的原生 API，由 EDA 自身处理坐标对齐、符号引用和文件格式。

已知可用的 MCP Server：
- KiCad：`kicad-mcp`、`kicad-mcp-pro`、`mcp-kicad-sch-api`
- 立创 EDA：`jlceda-mcp`

**回退规则：** 如果用户选择了 KiCad 或立创 EDA 格式，但当前环境没有对应的 MCP Server，提示用户安装对应 MCP Server，并自动回退到结构化连接表输出。

用户选择展示方式和需要覆盖的模块范围后，按选定方式输出。每个模块标注：器件型号、引脚连接、电源轨、关键参数值（电阻/电容值、电压/电流）。通过 Gate 6 后交付。

**输出文件：** `docs/hardware/07-schematics.md`（连接表/ASCII）或 `docs/hardware/07-schematics.d2`（D2）或 `docs/hardware/07-schematics.kicad_sch`（KiCad）或 `docs/hardware/07-schematics.json`（立创 EDA）

## 9. PDF 报告输出

所有阶段完成后，将 `docs/hardware/` 目录下的全部 markdown 文件合并输出为 PDF 报告。

脚本位于本技能目录下 `scripts/md_to_pdf.py`（即与 SKILL.md 同级的 `scripts/` 目录）。

```bash
python3 <skill_root>/scripts/md_to_pdf.py --merge docs/hardware/ docs/hardware/hardware-solution.pdf --title "项目名称-硬件方案报告" --theme green
```

可选配色方案（`--theme`）：
- `green`：工程翠绿（默认）
- `blue`：商务深蓝
- `gray`：极简石墨灰

生成前询问用户偏好的配色方案。

脚本依赖：`pip install weasyprint markdown`

PDF 包含封面（项目名称、日期）、自动分页、页眉页脚、表格样式。合并顺序按文件名排序（01-requirements.md → 06-decisions.md），确保阶段顺序正确。

**输出文件：** `docs/hardware/hardware-solution.pdf`
